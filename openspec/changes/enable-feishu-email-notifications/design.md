## Context

`notify.py` 已经包含飞书、邮件、企业微信和钉钉通道，并通过 `NOTIFY_CHANNELS` 的逗号分隔列表决定启用哪些通道。当前调度函数会遍历启用通道并返回每个通道的发送结果，这与“同时通知邮箱和飞书”的目标一致，但缺少明确的文档和测试来约束该行为。

这次变更应优先复用现有通道函数和调度结构，聚焦配置说明、失败隔离语义和测试覆盖，避免引入新的后台队列、异步执行或第三方依赖。

## Goals / Non-Goals

**Goals:**

- 支持用户通过 `NOTIFY_CHANNELS=feishu,email` 同时启用飞书和邮件通知。
- 保证多渠道发送时每个已知启用渠道都会被尝试。
- 保证单个渠道配置错误、发送失败或抛出异常时，不阻止其他渠道发送。
- 补充 `.env.example` 和 README 中关于飞书加邮件组合配置的说明。
- 增加标准库单元测试覆盖多渠道调度和失败隔离。

**Non-Goals:**

- 不新增通知渠道。
- 不改变飞书机器人或 SMTP 的协议实现。
- 不新增异步发送、重试队列或持久化投递记录。
- 不改变 Codex、Claude Code、Kimi 的通知入口协议。

## Decisions

- 继续使用 `NOTIFY_CHANNELS` 作为唯一的渠道启用入口。理由是它已经支持逗号分隔列表，用户只需配置 `feishu,email` 即可表达组合通知；替代方案是新增专用开关，例如 `ENABLE_EMAIL=true`，但会增加配置来源和优先级复杂度。
- 保持逐渠道串行发送。理由是当前通知数量很少，串行实现简单且便于返回稳定的结果摘要；替代方案是并发发送，但需要处理线程、超时聚合和测试复杂度，本变更收益不足。
- 保持“至少一个渠道成功则主流程成功”的现有退出语义。理由是通知脚本作为 hook 使用时，不应因为一个辅助渠道失败而让整体 hook 被视为失败；失败详情通过输出中的逐渠道状态暴露。若未来需要严格模式，可单独增加配置。
- 测试优先覆盖调度层，不直接调用真实飞书或 SMTP。理由是外部服务依赖凭据和网络环境，单元测试应通过替换 `CHANNEL_HANDLERS` 或构造假处理器验证行为。

## Risks / Trade-offs

- 邮件配置不完整会导致邮件通道失败，但飞书仍可能成功 -> 在文档中列出必填 SMTP 和收发件人配置，并在测试中覆盖失败隔离。
- 串行发送意味着慢通道会拖慢整体通知耗时 -> 继续依赖现有 10 秒网络超时，暂不引入并发复杂度。
- 未知渠道名会被忽略，用户可能误以为已发送 -> 文档中明确支持值为 `wecom`、`feishu`、`email`、`dingtalk`，并建议用脚本输出核对启用渠道结果。
