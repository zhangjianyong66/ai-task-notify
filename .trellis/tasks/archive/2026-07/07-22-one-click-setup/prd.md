# 设计一键配置功能

## Goal

为任意 Linux + Bash 用户提供可重复、可验证、可回滚的 Codex 通知一键配置入口，降低首次安装、换机、仓库迁移和升级修复时的配置遗漏风险，同时不泄露通知凭据、不覆盖无关用户配置，并始终保留官方 Codex 绕过通道。

## Background

- 当前部署需要分别维护项目 `.env`、用户级 `config.toml` 与 `hooks.json`、wrapper shim、脚本权限和 Bash PATH，仓库只有手工说明。
- 现有三入口架构保持不变：Codex 原生 `notify` 处理完成事件，`PermissionRequest` hook 处理审批事件，wrapper 只补齐用户提问和上游最终失败。
- 非托管 hook 必须由用户进入 `/hooks` 审核。Codex 没有公开稳定的逐 hook 信任查询接口，因此脚本不能可靠代替或验证该人工安全决策。

## Requirements

### R1. 平台与入口

- MVP 只支持 Linux + Bash 和 Codex，不自动配置 Claude Code 或 Kimi CLI。
- 对外统一使用 `./setup.sh install|check|uninstall`；shell 入口可委托内部 Python 标准库实现，用户无需直接调用内部模块。
- 项目路径、`HOME`、`CODEX_HOME` 和状态路径必须在运行时解析，不得硬编码用户名或克隆目录。

### R2. 安装模式

- `install` 默认运行交互式向导；重复运行兼作升级与修复。
- `install --non-interactive` 只使用已有 `.env` 或进程环境变量。缺少必填字段时必须在任何写入前失败，只能输出缺失字段名。
- 凭据不得作为命令行参数传入；非交互安装不得发送测试通知。
- `install --dry-run` 和 `uninstall --dry-run` 只报告计划，不写文件、不创建备份、不采集凭据、不联网。

### R3. 通知渠道配置

- 安装器支持交互式选择现有 `wecom`、`feishu`、`dingtalk`、`email` 渠道并生成或更新 `.env`。
- webhook、签名密钥和 SMTP 密码必须隐藏输入；所有输出只显示字段名及“已配置/缺失”状态。
- 已有完整 `.env` 默认复用。用户主动重新配置时，敏感字段可回车保留原值；未启用渠道只从 `NOTIFY_CHANNELS` 移除，其凭据、未知字段和注释默认保留。
- 生成或修改后的 `.env` 权限必须为 `0600`；进程环境中的凭据可用于校验，但不得自动复制到 `.env`。

### R4. Codex 与 shell 接入

- 安装器管理 `config.toml` 的顶层 `notify` 与 `log_dir`、`hooks.json` 的本项目 `PermissionRequest` handler、wrapper shim 以及 Bash PATH。
- 修改结构化配置时必须保留无关 TOML 内容、JSON hook 语义和 shell 内容；不支持的目标键写法必须安全失败，不能整文件模板覆盖。
- shim 必须位于独立目录并指向当前仓库 wrapper，不能覆盖官方 Codex。`~/.bashrc` 只新增带明确起止标记的受管 PATH 区块；已有未标记但等价的配置保持外部所有权。
- 安装完成后提示打开新 shell，不尝试改变父 shell 环境。

### R5. 依赖发现与检查

- Codex CLI 和 Python 是前置依赖；安装器只发现真实 Codex、排除 wrapper 自身、校验 `0.144.5+` 并报告最终入口，不调用 npm、Codex update 或 `sudo`。
- `check` 必须保持只读和离线，至少检查 Python、Codex 版本与入口、通知渠道完整性、`.env` 权限、Codex 配置、hook、shim、脚本权限、PATH 顺序及状态清单。
- `/hooks` 审核属于安装后的用户行为。自动配置完成即返回成功；安装器只提示该动作，不探测、不写入、不绕过 Codex 内部信任状态。

### R6. 幂等、冲突与事务

- 等价配置视为成功且不得重复写入、追加 PATH 或注册 hook。
- 交互安装发现冲突时必须逐项说明并确认；拒绝任一冲突时不执行任何写入。非交互安装遇到冲突必须在写入前失败。
- 修改现有目标前必须创建带时间戳、权限受控的备份；候选内容通过 TOML、JSON 或 Bash 校验后才能原子替换。
- 任一步失败时必须恢复本次已修改目标并移除本次新建项。输出可以报告备份路径，但不能显示备份内容或潜在敏感配置值。

### R7. 状态、迁移与卸载

- 每个用户只允许一个激活副本。其他路径的交互安装必须经确认迁移；非交互安装只有显式 `--migrate` 才可迁移。
- 迁移统一更新受管绝对路径，但不复制旧仓库 `.env`；新仓库必须已有有效配置或重新输入。
- 权限为 `0600` 的版本化状态清单记录受管项、摘要与恢复基线，但不得保存通知凭据正文。重复安装不得改变首次接管前的恢复基线。
- 交互式 `uninstall` 必须先展示计划且默认不继续；`--non-interactive` 视为明确授权。
- 卸载仅在当前值仍等于本项目受管值时恢复原值或移除新建项。安装后被用户修改的冲突项必须保留并导致非零退出；`.env`、普通备份和未标记外部 PATH 默认保留。

### R8. 测试通知

- 交互安装完成后可以询问是否向所有已启用渠道发送一次真实测试通知，默认不发送，只有明确确认后才能联网。
- 测试必须逐渠道报告结果。发送失败不回滚已提交的本地配置，但命令返回非零。

## Acceptance Criteria

- [ ] AC1（R1、R4）：全新 Linux + Bash 临时 HOME 可通过 `./setup.sh install` 完成 Codex 接入，所有生成路径来自当前环境，官方 Codex 入口仍可直接运行。
- [ ] AC2（R2、R6）：连续运行两次安装后配置语义和文件内容保持等价，不出现重复 hook、shim 或 PATH 项。
- [ ] AC3（R2、R3）：非交互配置完整时可无人值守安装；配置不完整时零写入失败，输出不包含任何配置值。
- [ ] AC4（R3）：交互向导支持多渠道、已有配置复用和安全重配；保留未知 `.env` 内容，落盘权限为 `0600`。
- [ ] AC5（R4、R6）：已有无关 TOML、hook 和 shell 配置在安装、修复及卸载后保持语义不变；复杂或冲突目标不会被静默覆盖。
- [ ] AC6（R5）：`check` 对全部关键目标给出机器可判定结果，保持文件、权限、链接和外部系统状态不变，也不把 `/hooks` 后续审核计为失败。
- [ ] AC7（R6）：候选配置无效或任意写入阶段失败时，本次变更完整回滚，不留下无效 TOML、JSON、Bash 文件或半安装 shim。
- [ ] AC8（R2、R6）：两个 `--dry-run` 能准确报告实际计划，且不创建状态、备份、临时文件或外部通知。
- [ ] AC9（R7）：跨路径安装不会产生多个激活副本；授权迁移可更新全部受管路径且不复制或输出旧凭据。
- [ ] AC10（R7）：卸载可恢复首次安装前状态；未经确认不执行，用户后改项不覆盖、不删除并返回非零，`.env` 默认保留。
- [ ] AC11（R8）：未明确确认时所有安装与检查路径均不发送通知；确认后逐渠道报告，测试失败保留本地安装并返回非零。
- [ ] AC12（全部）：自动化测试使用临时目录、fake Codex 和 mock 通知，不访问真实 webhook、SMTP、用户日志、用户配置或真实 Codex 会话。

## Out Of Scope

- 绕过或自动写入 Codex hook 信任状态，或改变用户审批策略。
- 改变现有通知事件架构、入口职责或事件协议。
- 自动安装或升级 Codex CLI、Python、Node、npm 或系统包。
- macOS、Windows、Zsh、Fish、Claude Code 和 Kimi CLI 的自动配置。
- 在多个仓库副本之间复制 `.env`，或删除用户凭据和普通备份。
- 引入桌面 GUI、第三方运行时依赖或常驻服务。
